# Delegate Implementation

実装をサブエージェントに切り出し、メインセッションは設計・監査・レビューに専念する。

節約の仕組み: メインで実装するとファイル閲覧・ビルドログ等のツール出力が会話コンテキストに蓄積し、以降の全ターンで読み直されるが、委譲すればエージェントの最終報告だけが残る。よってメインでは大きなファイルを読まない・大きなログを流さないことが節約の本体。

`$ARGUMENTS` があればそれを対象タスクとする。なければ進行中のタスクに適用する。
このスキルが定めるのは実装の担当分けのみ。イシュー・ブランチ運用などプロジェクトの開発フローは通常どおり従う。

## 役割分担

| 担当 | 作業 |
|------|------|
| メインセッション | 設計・作業分割、各単位の仕様策定、成果物の監査（diff レビュー）、統合レビュー |
| opus エージェント | 実装判断を伴う作業（新規機能、複数ファイル改修、エラーハンドリング設計） |
| sonnet エージェント | 仕様が固まった定型作業（既存パターン踏襲のテスト、ボイラープレート、リネーム、機械的変更） |
| Explore エージェント | 設計前のコードベース調査（読み取り専用）。ファイルダンプをメインに入れず結論だけ受け取る |

例外として、難易度が特に高い箇所（並行処理の設計、後方互換が微妙な変更、データ破壊リスクのある変更など、失敗コストが大きく設計と不可分な箇所）はメインセッションが直接実装してよい。その場合は理由を一言添える。

## ワークフロー

### 1. 設計（メインセッション）

1. コードベース調査は Explore エージェントに委譲する
2. 実装を独立した作業単位に分割し、各単位に担当（sonnet / opus / メイン直接）を割り当てる
3. 分割と割当を簡潔にユーザーへ報告してから着手する

### 2. 委譲（作業単位ごと）

- 原則 1 単位ずつ直列で進め、間に監査を挟む（「複数タスクを同時並行で進めない」原則）。ファイル衝突がなく完全に独立した単位のみ、その旨を伝えたうえで並列可
- `Agent` ツールで `subagent_type: "general-purpose"` ＋ `model: "opus"` または `"sonnet"` を指定する
- **fork は使わない**: fork はメインと同じモデルで動き `model` 指定が無効のため、委譲してもトークン節約にならない
- サブエージェントはメインの会話履歴・memory を読めないため、プロンプトを自己完結させる:
  - 目的と完了条件（build / vet / test が通ること）
  - 対象ファイルの絶対パスと、模倣すべき既存実装の場所
  - 適用すべき規約（プロジェクト CLAUDE.md の該当箇所、memory 由来の規約、呼ぶべきスキル名）
  - 禁止事項: git 操作（add / commit / push / branch）をしない、仕様外のファイルを変更しない
  - 報告フォーマット: 変更ファイル一覧＋変更要約＋テスト結果のみ。ファイル全文を返させない

### 3. 監査（単位ごとにメインセッション）

1. `git diff` で成果物を直接確認する。エージェントの自己申告を信用しない。巨大な diff は `git diff --stat` で当たりを付けてから要点を読む
2. 仕様との乖離・規約違反・仕様外変更の混入をチェックする
3. 修正が必要なら `SendMessage` で同じエージェントに差し戻す（エージェント側のコンテキストが維持される）。説明を書くより直す方が速い軽微な修正はメインで直す
4. 差し戻しは同一エージェントに最大 2 回。収束しなければ担当を一段上げる（sonnet → opus → メイン直接）

### 4. 検証・統合レビュー（メインセッション）

1. build / vet / test を実行する。出力が大きくなる場合は sonnet エージェントに「実行して失敗のみ要約」させる
2. 全単位の統合後に整合性（インターフェースの噛み合い、命名の一貫性、余計な変更の有無）をレビューする
3. 完了報告には各単位の担当モデルと監査結果を含める

## 注意

- エージェントが null を返した場合（ユーザースキップ・API エラー）は再委譲するか、メインで実装する
- 大規模一括変更で本格的な並列オーケストレーションが必要な場合は、トークン消費が増える旨を添えて Workflow の利用をユーザーに提案する（勝手に使わない）
