# 共通設定

- 必ず日本語で対応する
- 得意領域:
  - Go、Google Cloud（Cloud Run / Cloud Run Jobs / Secret Manager / Cloud Build）、バックエンド設計
  - Ruby, Rails（v8 / Hotwire / Turbo / Stimulus）
  - TypeScript（Next.js / Remix / NestJS）
  - フロントエンド: htmx + Alpine.js + Go templates + Tailwind（必要に応じて Next.js / Remix）

- 間違った指示は指摘し、より良い代替案があれば提案する
- 曖昧な指示や不明点がある場合は、確認してから進める
- 確信が持てないときは無理に実行せず、必ず確認する

## コーディング方針

- コードへの不必要なコメントは避ける
- 使用していないコードがあれば削除する
- Go のエラーメッセージは「failed to…」を避け、現在形・コロン区切りのチェーンで書く（『実用Go』準拠）
- Go のコンストラクタは `NewX` パターン、エクスポート/非エクスポートの境界を意識する
- Ruby は RuboCop（`~/dotfiles/.rubocop.yml`）の規約に従う

## 開発原則

- KISS: 最もシンプルな解決策を選ぶ。複雑さは必要になったときに追加する
- YAGNI: 現時点で必要な機能だけを実装する。「将来使うかも」で作らない
- DRY: 同じロジックが3箇所以上に現れたら共通化を検討する（2箇所は許容）
- SOLID: 設計時に適用する（特に単一責任の原則を重視）
- MECE: 選択肢の提示や分析では、漏れなくダブりなく整理する

### アーキテクチャ

#### Go
- レイヤードアーキテクチャ（repository / service / handler）を基本とする
- クリーンアーキテクチャの依存方向（内側に向く）を守る
- 周囲の似た実装を探し、それを参考にする
- Goの慣習に従う（例: エラーは現在形で、パッケージ名は短く、インターフェースは `-er` で終わるなど）
- エラーは適切にラップして伝播し、必要に応じてスタックトレースを含める（例: `fmt.Errorf("failed to do X: %w", err)`）
- 構造体のフィールドは必要に応じてエクスポート/非エクスポートを使い分ける。特に、外部からアクセスされないフィールドは非エクスポートにする
- コンストラクタ関数は `NewX` パターンを採用し、必要な初期化を行う。特に、依存関係の注入や不変条件の検証を行う
- Goの標準ライブラリや一般的なパッケージ（例: `context`, `errors`, `net/http`, `database/sql`）を積極的に活用する

#### Ruby
- MVC を基本とする
- Fat Model / Skinny Controller を心がける
- 周囲の似た実装を探し、それを参考にする
- Rails の慣習に従う（例: スコープはモデル、ビジネスロジックはサービスクラス、複雑なクエリはクエリオブジェクトなど）
- ActiveRecord の機能を活用し、可能な限り SQL を直接書かない
- エラーは適切に処理し、ユーザーにわかりやすいメッセージを提供する。特に、例外は rescue でキャッチして適切なレスポンスを返す
- コードの構造を整理するために、適切なディレクトリ構成を採用する（例: `app/models`, `app/controllers`, `app/services`, `app/queries` など）
- コードの再利用性を高めるために、共通のロジックやユーティリティ関数は別のファイルやモジュールに切り出す
- RSpec を使用して、モデルのバリデーションやサービスクラスのロジックなど、重要な部分のテストを作成する
- RailsのバージョンやGemの依存関係に注意し、適切なバージョン管理を行う（例: `Gemfile` と `Gemfile.lock` の管理）
- Railsのセキュリティベストプラクティスに従い、SQLインジェクションやクロスサイトスクリプティングなどの脆弱性を防止するための対策を講じる

#### TypeScript
- フレームワークの慣習に従う（Next.js なら API Routes / getServerSideProps、NestJS なら Controller / Service / Module など）
- 周囲の似た実装を探し、それを参考にする
- 型定義はできるだけ厳密にする（特に外部APIのレスポンスや関数の引数・戻り値など）
- インターフェースや型エイリアスを適切に使い、コードの可読性と保守性を高める
- ユーティリティ型（Partial, Pick, Omit など）を活用して、冗長な型定義を避ける
- 非同期処理は async/await を基本とし、Promiseチェーンは避ける
- エラー処理は try/catch で行い、適切なエラーメッセージを提供する
- コードの構造を整理するために、適切なディレクトリ構成を採用する（例: `src/controllers`, `src/services`, `src/models` など）
- コードの再利用性を高めるために、共通のロジックやユーティリティ関数は別のファイルやモジュールに切り出す

## ワークフロー

### 設計フェーズ

- 複数案をメリット・デメリット付きで MECE に比較し、推奨案を1つ選ぶ
- 推奨案が将来どういう変更で破綻するかも提示する

### 実装フェーズ

- 段階的に進める: 小さな変更を積み重ね、複数タスクを同時並行で進めない
- エラーは根本原因を特定してから対処し、解決してから次へ進む

### 調査フェーズ

- 調査プロセスをリアルタイムに報告する
- 結論をまとめる時は、重要な証拠も一緒に提示する

## 環境設定

- OS: WSL2（Ubuntu）on Windows
- エディタ: Vim（`~/dotfiles/.vimrc` / `.vim` 配下で管理）
- ターミナルマルチプレクサ: tmux
- Git 操作: tig を併用
- Docker 操作: lazydocker を併用
- 設定ファイルは `~/dotfiles` リポジトリで一元管理している（`bash ~/dotfiles/etc/install.sh` でセットアップ）
- `~/.claude` も dotfiles 管理下にある
- 新しい設定ファイルを追加・変更する場合は dotfiles リポジトリを更新する
- 秘密情報は `~/.bash_secrets` に分離し、リポジトリにはコミットしない

### Bash コマンド

- ファイル名・パス検索は `find` ではなく `fd`（Ubuntu では `fdfind`、エイリアスで `fd`）を使う
  - 例: `fd 'pattern' path/ --type f --exclude node_modules`
  - `find` でしか実現できない用途（`-newer`、複雑なパーミッション条件など）の場合のみ `find` を使う
- ファイル内容検索は `grep` ではなく `rg`（ripgrep）を使う
  - 例: `rg 'pattern' path/ --type go`
  - `grep` でしか実現できない用途や、パイプ経由でテキスト処理する場合のみ `grep` を使う
- ファイル閲覧は `cat` の代わりに `bat`（Ubuntu では `batcat`、エイリアスで `bat`）を使ってよい
- ファイルのコピーは `cp` より `rsync` を使う（シンボリックリンクの扱い、進捗表示、再試行機能などの利点があるため）
- インタラクティブな絞り込みは `peco` を使う（履歴・ブランチ・ファイル選択など）
- 「最近」「直近」など現在日時が必要な場合は、必ず `date` コマンドで取得
- `apt install` で新しいパッケージを入れた場合は、`~/dotfiles/etc/install.sh` にも追加する
- 言語ランタイム（Go / Ruby / Node など）は asdf で管理する。グローバルに直接入れず、`.tool-versions` を更新する
